# ADR-0002: JWT authentication (djangorestframework-simplejwt)

- **Status:** Accepted
- **Date:** 2026-07-10
- **Deciders:** Founding CTO

## Context

The API currently has no authentication — DRF defaults to `AllowAny`, so any
caller can list, upload, and download all evidence. This is the most urgent gap
before any shared use. The product is a **decoupled React SPA** today and will
likely add mobile/edge clients over five years. We need an auth mechanism that
serves multiple client types without server-side session affinity.

Options considered:
- **Session auth** — simplest for a single web console, but couples clients to
  Django sessions/CSRF and fits poorly with non-browser clients.
- **DRF TokenAuth** — trivial, but one static token per user, no expiry/refresh.
- **JWT (SimpleJWT)** — stateless, standard for SPA + multi-client, supports
  short-lived access tokens with refresh.

## Decision

Adopt **JWT via `djangorestframework-simplejwt`** with:
- short access-token TTL,
- refresh-token rotation,
- a token blocklist for revocation.

Set DRF defaults to `JWTAuthentication` + `IsAuthenticated`. **Authentication**
(who you are) lands in Phase 1; **authorization/tenancy** (what you may access)
is deliberately separated into Phase 2 (see ADR to follow / ARCHITECTURE.md),
because auth without tenant isolation is a false sense of security.

## Dependency justification (Engineering Bible §4)

- **Why needed:** stateless auth for a decoupled SPA and future clients.
- **Why existing insufficient:** no auth exists; DRF built-ins (session/token)
  don't fit the multi-client, stateless requirement as cleanly.
- **Maintenance cost:** JWT revocation is harder than sessions — mitigated by
  short TTL + rotation + blocklist. SimpleJWT is mature and low-churn.
- **Justified?** Yes, given the five-year multi-client horizon.

## Consequences

- **Positive:** stateless, horizontally scalable, client-agnostic auth.
- **Negative:** revocation requires the blocklist; token TTL/rotation must be
  configured correctly. Accepted and mitigated above.
- **Revisit if:** the product never gains non-browser clients *and* revocation
  overhead outweighs benefit — then reconsider session auth.
